# Hamburger Menu Implementation Plan

## Goal

Make the header (`MainNav.svelte`) responsive on mobile by replacing the current wrapping-link approach with a hamburger-triggered drawer menu built on bits-ui `Dialog`.

## Current State

- `MainNav.svelte` (`src/lib/components/navigation/MainNav.svelte:1`) renders a horizontal nav with Learn, Letters, Words, Practice, Sign in/out links + ThemeToggle
- On mobile (`≤640px` / `$bp-sm`), the layout stacks via `flex-direction: column-reverse` with `flex-wrap: wrap` — the layout breaks and looks clunky
- The nav links live inside a `.nav__cluster` div alongside ThemeToggle

## Approach

Use bits-ui `Dialog` to implement a slide-out drawer on the right side for mobile, while keeping the existing desktop layout unchanged.

**Why Dialog (not Sheet/Drawer)?**

- bits-ui v2.x has `Dialog` as the canonical overlay primitive. There's no separate `Sheet` or `Drawer` component.
- `Dialog` provides: `Root` (bind:open state), `Trigger` (hamburger button), `Portal` + `Overlay` + `Content` (the drawer panel), `Title` (accessibility), `Close` (close button).
- Built-in focus trap, ESC to close, ARIA attributes, keyboard navigation — all required for an accessible hamburger menu.

## Implementation Steps

### 1. Modify `MainNav.svelte`

Add a responsive hamburger menu inside the existing component. No new files needed.

**State:**

- `let mobileMenuOpen = $state(false)` — controls the Dialog open state

**Template changes:**

- Keep the desktop nav unchanged (`.nav__links` + `ThemeToggle`)
- Add a hamburger trigger button (`.nav__hamburger`) — visible only on mobile via CSS
- Wrap the existing `.nav__links` + `ThemeToggle` inside `Dialog.Content` as the drawer body
- Use `Dialog.Overlay` for the backdrop
- Use `Dialog.Title` (visually hidden but present for screen readers)
- Add a close button inside the drawer

**Layout strategy:**

```svelte
<nav class="nav">
 <div class="nav__inner container">
  <a class="nav__logo">...</a>

  <!-- Desktop: visible on screens > 640px -->
  <div class="nav__cluster nav__cluster--desktop">
   <div class="nav__links">...</div>
   <ThemeToggle />
  </div>

  <!-- Mobile: visible on screens ≤ 640px -->
  <Dialog.Root bind:open={mobileMenuOpen}>
   <Dialog.Trigger class="nav__hamburger">☰</Dialog.Trigger>
   <Dialog.Portal>
    <Dialog.Overlay class="nav__drawer-overlay" />
    <Dialog.Content class="nav__drawer">
     <Dialog.Title class="sr-only">Navigation Menu</Dialog.Title>
     <Dialog.Close class="nav__drawer-close">✕</Dialog.Close>
     <div class="nav__drawer-links">
      <!-- Same nav links as desktop -->
     </div>
     <ThemeToggle />
    </Dialog.Content>
   </Dialog.Portal>
  </Dialog.Root>
 </div>
</nav>
```

**Open question:** Should the nav links inside the drawer be duplicated from the desktop version, or should we extract them into a reusable snippet?

My recommendation: Extract a `{#snippet navLinks()}` snippet inside `MainNav.svelte` to avoid duplication. The snippet renders all `<a>` links (Learn, Letters, Words, Practice, Sign in/out). Use `{@render navLinks()}` in both desktop and mobile sections.

### 2. SCSS Styling

**Hamburger button (`.nav__hamburger`):**

- Hidden above `$bp-sm`, visible at and below
- Simple icon (text "☰" or SVG three-line icon)
- Styled to match the existing nav aesthetic (same height as nav\_\_inner, transparent bg)
- Focus-visible outline matching existing patterns

**Drawer (`.nav__drawer`):**

- Fixed position, right side, full height
- Width: ~280px (or 80vw max)
- Slides in from right (use CSS transform + transition, or rely on Dialog's built-in animations)
- Background: `var(--color-surface-nav)`
- Border-left: 1px solid var(--color-border)
- Padding: `$space-lg`
- Z-index handled by Dialog.Portal

**Drawer overlay (`.nav__drawer-overlay`):**

- Fixed inset-0, bg black/80 or similar
- Fade animation

**Drawer close button (`.nav__drawer-close`):**

- Positioned top-right corner
- Rounded, transparent bg, hover state

**Drawer links (`.nav__drawer-links`):**

- Vertical stack with larger touch targets (min 44px height)
- Same link text and active state logic as desktop
- Each link styled as full-width, with padding, border-radius
- No `.nav__count` badges? — Keep them, they're useful info

**Drawer ThemeToggle:**

- Centered or left-aligned at bottom of drawer
- Same component, no changes needed

**Responsive visibility:**

```scss
.nav__cluster--desktop {
 display: flex;
}
.nav__hamburger {
 display: none;
}

@media (max-width: $bp-sm) {
 .nav__cluster--desktop {
  display: none;
 }
 .nav__hamburger {
  display: flex;
 }
}
```

### 3. Close drawer on navigation

When a user taps a nav link inside the drawer, set `mobileMenuOpen = false` to close the drawer. This can be done by:

- Adding `onclick={() => mobileMenuOpen = false}` to each `<a>` tag inside the snippet, OR
- Wrapping the links in a click handler

Simplest approach: add `onclick={() => mobileMenuOpen = false}` to each link in the snippet (the snippet is only rendered inside the drawer). Or use a wrapper div with event delegation.

## Bits UI API Reference (from Context7 docs)

**Dialog.Root props:**

- `open: boolean` — bindable open state
- `onOpenChange: (open: boolean) => void`

**Dialog.Trigger**: The button that opens the dialog. Renders as `<button>`.

**Dialog.Portal**: Renders children into `<body>` using a portal.

**Dialog.Overlay**: Backdrop overlay. Gets `data-state="open"` / `data-state="closed"` attributes.

**Dialog.Content**: Main dialog panel. Gets `data-state` attributes.

**Dialog.Title**: Accessible title (required for screen readers).

**Dialog.Close**: Button that closes the dialog.
